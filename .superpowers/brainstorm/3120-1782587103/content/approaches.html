滚动同步方案对比

目标：编辑器视口中央的行 → 预览中对应的渲染内容

A

标题锚点内插法

用文档中的标题作为"锚点"：匹配 markdown 源码中的 # 标题与预览中渲染后的 h1~h6 元素，在锚点之间按行数线性内插。无标题文档退回到百分比。

B

渲染后逐块锚点注入法

渲染完成后，将 markdown 按空行分块，同时遍历预览 DOM 的直接子元素，逐块匹配并在预览中注入 data-line 属性。在每两个标记行之间的内容，按预览内的像素距离线性内插。

C

Rust 渲染器改造

修改 Rust 端的 render_markdown 函数，在 pulldown_cmark 输出中为每个非空行注入行号标记。最精确，但需要改 Rust 源码、重编译，且 pulldown_cmark 不直接暴露行号。

推荐 B — 渲染后逐块锚点注入

  • 纯 JS 实现，不需要改 Rust，迭代快
  • 逐块匹配比标题锚点更密集，精度更高
  • 即使文档没有标题也能工作（直接按块数量映射）
  • 实现复杂度适中

权衡

  • markdown 块和预览子元素并非严格 1:1 对应（列表/表格被包裹），匹配不够完美
  • 需要在 updatePreview 后注入标记，增加少量渲染开销
