方案 B 详细设计：渲染后块匹配 + 像素内插

纯 JS 端，不改 Rust，每 30ms 重建映射，实时跟随编辑

第一步：Markdown 源码分块 增量

按空行分块，跟踪围栏代码块（``` 内部不分割），每块记录其起始行号。

// 分块结果 [块1]
行 0 ─ # 第一章 引言
行 1 ─ 这是第一段文字...
// (空行 → 块边界)
行 3 ─ 第二段内容开始...
// (空行 → 块边界) [块2]
行 5 ─ ## 方法
行 6 ─ 实验采用...
// (空行 → 块边界) [块3]
行 8 ─ ```python
行 9~12 ─ 代码块内部，不拆分
行 13 ─ ```

第二步：DOM 子元素映射 增量

收集预览中所有块级子元素（P, H1~H6, PRE, UL, OL, TABLE, BLOCKQUOTE, HR），按数量比例映射到 markdown 块。

块索引 0
<h1> 第一章 引言
<p> 这是第一段文字...
块索引 1
<p> 第二段内容开始...
块索引 2
<h2> 方法
<p> 实验采用...
块索引 3
<pre><code> python...

第三步：行号 → 预览位置映射 增量

每个块元素注入 data-md-line 属性，记录其 offsetTop，构建 [{line, top}, ...] 数组。

第四步：滚动同步 复用

替换现有字符比例逻辑：

  • 编辑器→预览：视口中央行号 → 二分查找包围它的两个标记 → 像素内插 → 预览 scrollTop
  • 预览→编辑器：视口中央像素 → 二分查找包围它的两个标记 → 行号内插 → 编辑器居中
  • re-render 恢复：同上，在 updatePreview 最后执行

关键特性

  • 不改 Rust，不改 markdown 源码
  • 每次 updatePreview（30ms debounce）重建完整映射
  • 标记在渲染后的 DOM 上，不干扰解析
  • 无标题文档也能工作（空行分块策略不依赖标题）
  • 代码块、表格等复杂结构通过比例映射自然处理

局限

  • 块数量和 DOM 子元素数量不等时的比例映射精度
  • 巨型文档（>10K 行）块匹配有线性开销，但 DOM 子元素不会很多
